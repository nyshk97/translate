# CLAUDE.md

## プロジェクト概要

自分専用の macOS ネイティブ翻訳ツール。メニューバー常駐 + グローバルショートカットで呼び出すランチャー1画面のアプリ。万人向けではなく、自分の用途（当面は英↔日中心）に振り切って作る。

## 最優先の設計方針

- **速度は「初動」を最優先する。** 総生成時間ではなく、「ショートカットを押した瞬間に最初のトークンが出てくる」体感を最大化する。
- モデルの TTFT（最初のトークンが返るまでの時間）が全体の支配項であり、そこ自体はモデル選定以外で動かせない。だから初動までの周辺をすべて削る方向で設計する。
- 汎用性より自分の用途への最適化を優先する。迷ったら速くてシンプルな方を選ぶ。

## 非目標（やらないこと）

先回りで作らない。これらを「将来のため」に実装し始めたら止める。

- Web 版を作らない
- 多言語 UI・多言語対応をやらない（当面は英↔日のみ）
- 中間サーバーを立てない
- 認証・課金・レートリミット・複数ユーザー対応をやらない
- 重い汎用抽象化やプラグイン機構を入れない

## 機能スコープ

### 必要（コア）

- ショートカットでどのアプリからでも翻訳（テキスト選択 → ショートカット。選択テキスト取得に Accessibility 権限が要る）
- 日本語に戻してチェック（翻訳経路を逆向きに1回呼ぶだけ）
- スクショ翻訳（画面範囲選択 → vision モデル）
- ローカル完結の履歴（SQLite かファイル）

### あると嬉しい（後回し可）

- ニュアンスをサッと調整（翻訳後にボタンで再翻訳）
- トーン別に複数案を提案（デフォルトにはしない。ボタンで明示的に出す）
- 長文を要約（別アクション）
- 画像をすぐに翻訳（スクショ翻訳と同じ vision 基盤の副産物）

### 不要（作らない）

添削・校正 / あとから内容を追加 / 発音再生 / 文体・シーン選択 / 自分専用の文体タグ / AI 返信作成 / 会話のラリー翻訳 / ショートカット添削 / URL 翻訳 / 法人・チーム / 用語解説

### 実装メモ

- 新しく基盤が要るのは2つだけ: (1) vision 経路（スクショ・画像）、(2) 選択テキスト取得（Accessibility 権限）。残り（戻し訳・履歴・ニュアンス調整・複数案・要約）は、テキスト翻訳の経路にプロンプト違いを足すだけで実装できる。
- vision 経路はテキスト経路と別系統として扱う。vision は TTFT が大きく遅いので、「初動最速」方針はテキスト経路にのみ適用し、画像・スクショは「待つ前提」の別トレードオフで設計する。

## アーキテクチャ

- **プロバイダー API を直叩きする**（アプリ → プロバイダーの1ホップ）。中間サーバーは置かない。
- API キーは Keychain に保存する。コードや設定ファイルに平文で置かない。
- ネイティブの HTTP（`URLSession`）でリクエストする。ブラウザ由来の CORS / プリフライト（OPTIONS）は発生しないので、その対策は不要。
- 起動時および常駐中に軽いウォームアップリクエストを撃ち、TLS / HTTP2 のコネクションを温めておく。初回翻訳のハンドシェイク往復を消す。
- ストリーミング応答を逐次表示する。

## モデル選定・ルーティング

- 翻訳・言語判定のデフォルトは **TTFT の小さいモデル**（例: Groq 上のモデル / Gemini Flash 系）を使う。
- 要約・詳しい解説など、品質が要る重い処理だけ高性能モデルに振り分ける。
- プロンプトは多言語・多条件の分岐なしの **短い固定文** にする。入力トークンを最小化して、プレフィルと TTFT を縮める。

## 翻訳方向と言語判定

- 入力が日本語 → 英語へ翻訳する。
- 入力が日本語以外（英語を含む主要言語）→ 日本語へ翻訳する。
- 言語判定は **ローカルの文字種判定** で行い、LLM は呼ばない。ひらがな・カタカナを含めば日本語入力とみなして英訳、含まなければ和訳する。これで言語判定の1往復を省き、初動を速くする。
- 「主要言語 → 日本語」は出力が常に日本語なので、入力言語の厳密な特定は不要。和訳プロンプト一本で LLM が入力言語を吸収する。判定ロジックが必要なのは「日本語か否か」の2値だけ。

## 想定スタック

- Swift / SwiftUI（必要に応じて AppKit）
- メニューバー常駐（`NSStatusItem`）+ グローバルショートカットで `NSPanel` を即表示するランチャーモード
- 履歴はローカル保存（SQLite かファイル）。クラウドには保存しない。

## UI / UX 方針

- 1画面で完結。ショートカット → ウィンドウ即表示 → 入力 → 即送信。
- ストリーミング描画は数十 ms ごとにバッファしてから state を更新する。トークンごとの再描画で SwiftUI がカクつくのを避ける。
- 操作をブロックしない。送信中でも次の入力・中断ができるようにする。
- 説明文に頼らず、デフォルト挙動だけで快適に使えるようにする。

## コーディング指針

- 依存は最小限。速度に効かないライブラリ・抽象化は入れない。
- 非アクティブ時はバックグラウンド処理を止め、CPU / メモリを抑える（常駐アプリのため）。
- 新機能を足すときは、既存の初動速度に影響がないか毎回確認する。
